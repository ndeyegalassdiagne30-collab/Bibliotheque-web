Concepts fondamentaux de collaboration

Fork

Opération qui crée une copie indépendante d'un dépôt sur votre compte GitHub, tout en conservant l'historique complet des commits.

Cas d'usage typique

# Situation : Contribuer à React (projet Facebook)
1. Visitez : github.com/facebook/react
2. Cliquez sur le bouton "Fork" (en haut à droite)
3. Sélectionnez votre compte utilisateur
4. Résultat : github.com/votre-compte/react

# Le fork crée une copie complète :
- Historique des commits préservé
- Lien vers l'original conservé
- Possibilité de synchroniser les mises à jour
Open Source Contribution

Avantages : Permet de travailler librement sans affecter le projet original, idéal pour proposer des modifications via Pull Request.

Pull Request (PR)

Mécanisme de proposition de modifications permettant de demander l'intégration de vos changements dans la branche principale d'un projet.

Processus complet

# 1. Après avoir poussé vos modifications
git push origin feature/ma-fonctionnalite

# 2. Sur GitHub, créer la Pull Request
- Aller dans l'onglet "Pull requests"
- Cliquer "New pull request"
- Sélectionner : base: main <- compare: feature/ma-fonctionnalite

# 3. Remplir le template
Titre: [Feature] Ajout système d'authentification
Description:
- Implémentation de JWT
- Tests unitaires ajoutés
- Documentation mise à jour

# 4. Assigner des reviewers
# 5. Ajouter des labels (enhancement, needs-review)

Bonnes pratiques : PR petites et focalisées, description claire, liens vers les issues concernées.

Code Review

Processus d'examination du code par les pairs pour garantir la qualité, détecter les bugs et partager les connaissances.

Exemple de revue constructive

// Fichier: auth.js - Ligne 42
function validateUser(input) {
    // AVANT: Risque de faille XSS
    return input; 
    
    // SUGGESTION: Nettoyer les entrées utilisateur
    return sanitizeHtml(input);
}

// Commentaire du reviewer :
// "Bonne logique globale ! 
//  Attention à la sécurité : suggestion d'ajouter 
//  une sanitization des entrées.
//  Voir OWASP Top 10 - A7:2017-Cross-Site Scripting"

// Réponse de l'auteur :
// "Correction effectuée ! Merci pour la vigilance."

Critères de revue : Architecture, sécurité, performance, tests, lisibilité, conventions.

Issues

Système de suivi intégré pour gérer bugs, fonctionnalités, tâches et discussions autour du projet.

Template d'issue bien structuré

---
Titre: [BUG] Erreur 500 sur page de connexion
---

**Description du bug**
Lors de la tentative de connexion avec des identifiants valides, une erreur 500 apparaît.

**Étapes pour reproduire**
1. Aller sur /login
2. Entrer email: user@test.com
3. Entrer mot de passe: correct123
4. Cliquer sur "Se connecter"
5. Voir l'erreur 500

**Comportement attendu**
Redirection vers le tableau de bord

**Environnement**
- Navigateur: Chrome 120
- OS: macOS 14
- Version de l'API: v2.1.0

**Logs d'erreur**
TypeError: Cannot read property 'id' of undefined
    at authenticate (auth.js:45)

**Labels** : bug, haute-priorité, auth

Types d'issues : Bug, Feature, Documentation, Discussion, Question, Tâche technique.

Workflows de gestion de branches

Git Flow

Workflow structuré pour projets avec cycles de release planifiés.

Branches principales

  • main - Code en production, version stable
  • develop - Intégration des fonctionnalités

Branches de support

  • feature/* - Nouvelles fonctionnalités
  • release/* - Préparation des releases
  • hotfix/* - Corrections urgentes

Scénario complet

# Développement d'une fonctionnalité
git checkout -b feature/paiement-stripe develop
# Travail sur la fonctionnalité...
git commit -m "Ajout intégration Stripe"
git push origin feature/paiement-stripe
# Merge dans develop après review
git checkout develop
git merge --no-ff feature/paiement-stripe

# Préparation d'une release
git checkout -b release/1.2.0 develop
# Tests, corrections de bugs mineurs
git commit -m "Fix: correction typo dans emails"
# Fusion dans main et develop
git checkout main
git merge --no-ff release/1.2.0
git tag -a v1.2.0 -m "Version 1.2.0"
git checkout develop
git merge --no-ff release/1.2.0

# Hotfix critique
git checkout -b hotfix/security-patch main
git commit -m "Fix: faille de sécurité critique"
git checkout main
git merge --no-ff hotfix/security-patch
git tag -a v1.2.1 -m "Patch sécurité urgent"
git checkout develop
git merge --no-ff hotfix/security-patch

Quand l'utiliser : Applications avec versions, maintenance multiple versions, équipes structurées.

GitHub Flow

Workflow simplifié pour déploiement continu.

Branche unique

  • main - Toujours déployable

Branches temporaires

  • feature/* - Fonctionnalités
  • fix/* - Corrections

Cycle de développement continu

# 1. Créer une branche depuis main
git checkout main
git pull origin main
git checkout -b feature/ajout-recherche

# 2. Développer avec commits réguliers
git add .
git commit -m "Ajout composant barre recherche"
git commit -m "Intégration API Elasticsearch"
git commit -m "Tests unitaires pour recherche"

# 3. Pousser et créer Pull Request
git push origin feature/ajout-recherche
# Sur GitHub : créer PR vers main

# 4. Phase de review
# - Discussions sur le code
# - Modifications demandées
# - Approbation

# 5. Tests automatisés (CI)
# - Linting
# - Tests unitaires
# - Build

# 6. Merge et déploiement
# Après merge automatique dans main
# Déploiement automatique en production
# Suppression de la branche

# 7. Monitoring
# Surveiller les métriques post-déploiement
# Prêt pour itérer

Idéal pour : Startups, SaaS, projets web, équipes agiles, déploiements fréquents.

Conventions de nommage des branches

Standards de nommage

Feature branches

feature/authentification-jwt
feature/42-panier-achat
Nouvelles fonctionnalités, format: feature/description-courte

Bugfix branches

bugfix/correction-login
bugfix/123-erreur-paiement
Corrections de bugs, format: bugfix/issue-description

Hotfix branches

hotfix/securite-urgente
hotfix/1.2.1-patch-critique
Corrections urgentes en production

Release branches

release/1.2.0
release/v2.0.0-beta
Préparation des versions

Chore branches

chore/mise-jour-deps
chore/refactor-documentation
Tâches de maintenance

Experiment branches

experiment/nouveau-design
wip/algorithme-optimise
POCs et expérimentations

Règles d'or

  • ✓ Une branche = un objectif
  • ✓ Noms en minuscules
  • ✓ Utiliser des tirets (-)
  • ✓ Inclure numéro d'issue si possible
  • ✓ Descriptif mais concis
  • ✓ Supprimer après merge

Bonnes pratiques et recommandations

Pour les Pull Requests

  • ✓ PR de taille raisonnable (max 400 lignes)
  • ✓ Description claire du contexte
  • ✓ Screenshots pour les UI
  • ✓ Liens vers les issues concernées
  • ✓ Checklist de validation

Pour les Reviews

  • ✓ Commentaires constructifs
  • ✓ Suggérer, pas imposer
  • ✓ Vérifier les tests
  • ✓ Attention à la sécurité
  • ✓ Valider dans les 24h

Pour les Issues

  • ✓ Templates standardisés
  • ✓ Labels pour catégoriser
  • ✓ Milestones pour planifier
  • ✓ Assigner clairement
  • ✓ Fermer avec # dans commit
